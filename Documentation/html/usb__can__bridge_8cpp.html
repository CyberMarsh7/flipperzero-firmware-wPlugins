SLCAN command extraction.

This function updates next_command pointer to point at the beggining of next SLCAN command. This function performs the following actions:

  1. if *next_command is null (new CDC frame)
    • set next_command to point the first byte of USB CDC (VCP) frame.
  2. Otherwise:
    • go to next command by updating next_command pointer : it consists to move pointer at the end of previous command by adding cmd_length value ( cmd_length is used like an index).
  3. reset cmd_length to 0
  4. search next CR or LF USB character (relative to next_command pointer) in CDC frame, incrementing each time cmd_length which is used as an index.
  5. if CDC frame last index is reached before finding a CR or an LF character, returns false
  6. returns true otherwise after incrementing command_length a last time to include termination character (CR or LF) .
    Precondition
    next_command shall be either Null (new CDC frame) or shall be called with:
    • next_command pointer set to the last _usb_uart_bridge_get_next_command() returned value.
    • cmd_length pointer set to the last _usb_uart_bridge_get_next_command() returned value.
    Parameters
    [in,out]next_commandWhen passed to the function, it correspond to a pointer to last processed USB CDC frame chunk (SLCAN command). At the end of the function it points to next SLCAN command.
    [in,out]cmd_lengthWhen passed to the function, it correspond to the length of last processed USB CDC frame chunk (SLCAN command) pointed by next_command. At the end of the function it corresponds to the length of next SLCAN command.
    [in]cdc_dataPointer to current USB CDC frame to be parsed. The first time this fonction is called (for a new cdc frame), next_command is pointing to the same location than this parameter. Each time a valid command is parsed next_command pointer move to next command location in this frame.
    [in]cdc_frame_lenLength of current CDC frame.
    Returns
    True if it remains commands to process in current USB CDC frame, False otherwise (next CDC frame is to be read).

SLCAN S<X> command processing : CAN bus configuration.

This function performs the following actions:

  1. Check S command length is 3 (S<X>\r). If it is not the case sends an error message on VCP (USB CDC channel) via M_USB_CAN_ERROR_INVALID_LENGTH() before returning.
  2. Check <X> is a digit. If it is not, sends an error message on VCP (USB CDC channel) by calling usb_can_bridge_error() before returning.
  3. configure CAN bus through mcp2518fd::begin() function. bitrate_table is used to map S<X> command parameter to mcp2518fd::begin() speedset parameter.
  4. set can_if_initialized to true, in order to be able to process Tx commands (through usb_can_bridge_send()) normally.
    Precondition
    data parameter shall be an S command, or shall begin like an S command (with an 'S') : "Sxxxxxxxxx"
    Parameters
    [in]usb_canApplication Control logic related data. It is used because it contains can driver instance (mcp2518fd)
    [in]dataSLCAN S command.
    [in]lencommand length (data string length).

SLCAN Transmit command processing (txxx,Txxxx,rxxxx,Rxxxx)

This function performs the following actions:

  1. set header input format to be provided to sscanf pasring function according to SLCAN command processed:
    • t%3x%1x for tiiil command
    • T%8x%1x for Tiiiiiiiil command
    • r%3x%1x for riiil command
    • R%8x%1x for Riiiiiiiil command
  2. set max identifier valid value according to the SLCAN command : 1FFFFFFF for extended frames (command Tiiii... and Riii...), 0x7ff otherwise
  3. set command length bounds according to the SLCAN command : 5 < length < 22 for standard frames (tiii..,riii...), 10 < length < 27 for extended frames.
  4. check can_if_initialized is true, in order to ensure CAN interface have been initialized (via usb_can_bridge_configure). If it is not he case send an error message before returning via usb_can_bridge_error.
  5. check SLCAN command length is within the previously defined bounds. If it is not the case transmit error message before returning via M_USB_CAN_ERROR_INVALID_LENGTH.
  6. parse command header (from index 0 to min length) in order to get CAN ID and CAN data length code (DLC) through sscanf. The format string computed in step1 is used.
  7. determine length of data field : 0 if this is an RTR (riiil,Riiiiiiiil), =DLC for Tx (tiiildddd...,Tiiiiiiiilddddd...)
  8. check DLC field value is consistent with SLCAN command length it shall be equal to header length (5 or 10 + data field length)
    • If it is not the case send error message "[err]: can payload length discrepency : expected <X> got <Y>" through usb_can_bridge_error before retruning
  9. check if ID field value is valid (ie. inferior to value computed previously)
  10. switch on Tx LED.
  11. add previously computed data field length to Tx counter
  12. parse and extract SLCAN command data field
  13. transmit CAN frame through mcp2518fd::sendMsgBuf() with previously extracted data, CAN ID and dlc fields
  14. call timer to schedule Tx LED switch off.
Precondition
data parameter shall be a Tx command, or shall begin like an Tx command (with a 'T', a 't', an 'R', or an 'r' )
Parameters
[in]usb_canApplication Control logic related data. It is used because it contains can driver instance.
[in]dataSLCAN Tx command (riiil,Riiiiiiiil,tiiildddd...,Tiiiiiiiilddddd...)
[in]lencommand length (data string length)
[in]extendedtell if CAN frame to be sent is an extended frame or not (SLCAN command begins with an uppercase letter)
[in]rtrtell if CAN frame to be sent is a standard TX or an RTR (Remote Transmission Request) frame (i.e SLCAN command begins with a 'T/t' or with an 'R/r')

◆ usb_can_tx_thread()

static int32_t usb_can_tx_thread ( void *  context)
static

CAN TX thread : Process SLCAN commands send by the host.

This thread is started by usb_can_worker(). It is composed by aloop which performs the following actions:

  1. wait until a frame is received through USB VCP channel
  2. If Application is in USB loopback mode it will just send back received frame and wait next USB frame.
  3. Otherwise, if application is in USB-CAN-bridge mode it will process SLCAN commands in a loop as follows:
    1. get next command in USB CDC reception buffer (delimited by next carriage return) by calling _usb_uart_bridge_get_next_command()
    2. either call :
      • usb_can_bridge_configure() if it is a configuration Command (Sx)
      • usb_can_bridge_send() if it is a transmit command (tiiildddd...., Tiiiiiiiilddddd..., riiil..., Riiiiiiiil.)
      • usb_can_bridge_error() an send " [err]: command <command>><x> not implemented yet!"" if command is not implemented.
      • M_USB_CAN_ERROR_INVALID_COMMAND otherwise

◆ usb_can_worker()

static int32_t usb_can_worker ( void *  context)
static

application main thread : receives CAN frames and forward it to host.

This thread is started upon application mode selection by user. It perform following actions:

  1. if can is enabled (USB loopback test not selected)
    • initialize CAN
    • if CAN test is enabled : enter in an infinite transmit loop (identifier 0x7E57CA, data="CANLIVE")
  2. initialize USB Virtual Com port (USB CDC class)
  3. display Serma welcome banner on Virtual Com Port (VCP)
  4. print application mode (USB loopback or USB-CAN-Bridge)
  5. start usb_can_tx_thread() (usb_can->tx_thread) used for CAN transmission.
  6. start thread Main loop : This loop will block until either :
    • a packet is received on CAN Link : flag set via usb_can_on_irq_cb(). in this case:
      1. CAN frame data field is read via readMsgBuf()
      2. interrupt flags are cleared to deassert interrupt pin mcp2518fd_readClearInterruptFlags(). This can not be done in interrupt mode because SPI communication is needed.
      3. GPIO used for interrupt is re-enabled furi_hal_gpio_enable_int_callback()
      4. CAN ID is read through getCanId() and interpreted through isExtendedFrame()
      5. CAN frame is formated to SLCAN format ( identifier and <d> data)::
        • for standard frames : tiildddd...=> snprintf() format t%03X%1d%02X%02X...
        • for extended frames : TIIIIIIIIlddd... snprintf() format T%08X%1d%02X%02X ...
      6. formated SLCAN frame is sent on USB VCP through furi_hal_cdc_send().
    • VCP is configured via flipper buttons via usb_can_set_config()
